# 组件拓展层指南

> 8 种结构化表达的完整语法、示例和设计原则。
> SKILL.md 只保留组件概览表，本文档承载详情。

## 两种模式

```bash
# 默认模式（纯净）：直接转换,无组件无动画
node scripts/convert.mjs article.md --theme zhijian

# 组件模式：Agent 智能改写 + 开场动画 + 8 种结构化表达
# Agent 分析文章 → 提取组件候选 + 开场动画参数 → 向用户确认 → 改写 Markdown → convert 生成
/wechat-styler article.md --components
```

**组件模式下 Agent 的职责：**
- 读完整篇文章，识别适合用组件呈现的内容(表格→对比卡片、流程→步骤块、金句→金句块等)
- 从文章内容提取开场动画的标题/副标题/标签
- 向用户展示改写计划，等确认后执行
- 在内存中改写 Markdown(不改原文件)，然后调用 convert

默认模式下，所有组件语法会 fallback 成普通有序列表、无序列表或引用块，不会出现乱码或暴露标记。

## 8 种结构化表达

| 组件 | 语法 | 视觉形态 | 适用场景 |
|------|------|----------|----------|
| 金句块 | `> **核心观点**` | 居中 + 楷体大字 + 上下留白 | 文章核心论点、关键金句 |
| 轻量标记 | `==关键词==` | 底部 1px 暖陶色细线 | 正文中的关键词强调 |
| 提示块 | `> [!NOTE] 内容` | 结构色（墨蓝）前缀 + 同行正文 | 补充说明、技术旁注 |
| 警告块 | `> [!WARNING] 内容` | 强调色（暖陶）前缀 + 同行正文 | 注意事项、踩坑提醒 |
| 步骤序号 | `1. [step] 动作` | 加粗数字 01/02/03 + 正文 | 操作流程、动作清单 |
| 流程卡片 | `:::flow` 围栏 | section + flex 横排 + 箭头 | 输入→处理→输出、阶段流程 |
| 对比卡片 | `:::compare` 围栏 | section + flex 多栏 | 方案 A vs B vs C |
| 时间线 | `:::timeline` 围栏 | 竖向圆点 + 竖线 | 演进历程、项目复盘 |

## 组件语法示例

### 金句块

```markdown
> **员工效率不等于组织效能。**
```

### 轻量标记

```markdown
Rule 是唯一的主控。它不做检索，不做写作，只做==编排和决策==。
```

渲染效果：「编排和决策」底部出现 1px 暖陶色细线，文字保持正文色不变。像讲义边上的铅笔标记。

### 提示与警告

```markdown
> [!NOTE] 沉淀成 Skill 而不是文档，Skill 自带执行力。

> [!WARNING] 这三步的顺序不能颠倒。
```

### 步骤序号

```markdown
1. [step] 先对齐 90 天能看见结果的小目标。
2. [step] 选一个高频高价值的场景切入。
3. [step] 把跑通的过程沉淀成 Skill。
```

### 流程卡片

```markdown
:::flow
对齐目标 → 选场景试点 → 沉淀成 Skill
:::
```

### 对比卡片（**包裹** 表示高亮项）

```markdown
:::compare
**克制派 · 推荐** | wechat-styler | 沉淀在 Skill
组件派 | gzh-design | 沉淀在组件库
手工派 | 秀米 | 沉淀在 PPT
:::
```

### 时间线

```markdown
:::timeline
2026 · 03 | v1.0 初版 | 3 套基础主题
2026 · 07 | v1.5 组件拓展层 | 6 个结构化组件
:::
```

## 设计原则

1. **克制优于装饰**：组件不用色块，只靠排印手段（字号、字体、留白、细线）+ section 布局建立层次
2. **颜色从主题取**：所有组件的颜色/字体从当前主题的 yaml 配置取，各主题自动适配，不写死任何颜色
3. **横向卡片用 flex:1 等分 + overflow 兜底**：≤3-4 个卡片用 `flex:1` 等分宽度（内容自动换行，不滚动）；≥4-5 个才加 `overflow-x:auto` 横向滚动。`align-items:stretch` 保证卡片等高。参考 gzh-design-skill 实测方案。
4. **不用 table**：公众号编辑器对 `<table>` 会强制灰边框，但 SGP 这类行×列**矩阵对比**例外——矩阵信息密度高，表格比 compare 卡片更清晰，保留 markdown 表格
5. **卡片内容要短**：compare 卡片的 title 和 desc 用短句（1-2 行），不用大段文字。内容太长会把卡片撑宽，破坏 flex 等分效果
6. **默认沉默**：不传 `--components` 时，组件语法优雅降级成普通 markdown，不增加任何认知负担

## 横向卡片实现方案（v1.7.2 定稿）

### 核心策略：flex:1 等分优先，overflow 滚动兜底

简单 flex（`display:flex` + `align-items:stretch` + `flex:1`）在微信客户端是支持的，被挤压的问题出在卡片内容太长 + 固定宽度组合。最终方案：

**≤3 个 compare 卡片 / ≤4 个 flow 步骤**：
```css
/* 外层不滚动 */
<section style="display:flex;align-items:stretch;">
  /* 子元素等分，内容自动换行 */
  <section style="flex:1;min-width:0;">短内容</section>
  <section style="flex:1;min-width:0;">短内容</section>
</section>
```

**≥4 个 compare 卡片 / ≥5 个 flow 步骤**：
```css
/* 外层滚动 */
<section style="overflow-x:auto;-webkit-overflow-scrolling:touch;">
  <section style="display:flex;align-items:stretch;min-width:max-content;">
    /* 子元素固定宽度 */
    <section style="flex:0 0 170px;min-width:170px;">内容</section>
  </section>
</section>
```

### 为什么不用 inline-block 方案

inline-block + white-space:nowrap 方案虽然兼容性好，但有两个问题：
1. **高度不齐**：inline-block 无法自动等高，卡片长短不一时底部参差
2. **固定宽度导致长内容被拉宽**：`width:200px` + overflow 滚动，如果卡片 desc 是一整句话，单张卡片会被撑到很宽，滑动很久才看完

flex:1 等分让卡片各占 1/N 宽度，内容自动换行，既等高又不拉宽。只有卡片数量多时才切换到固定宽度 + 滚动。

### 卡片内容约束

卡片内容必须短——这是 flex:1 等分成立的前提：
- tag：2-5 字标签
- title：1 行标题（13px）
- desc：1-2 行说明（11px），**不用大段文字**
- 如果对比内容太长，不适合用 compare，保留 markdown 段落

## 组件模式智能改写规则

Agent 读文章后，按以下规则识别适合用组件呈现的内容：

| 原文模式 | 适合的组件 | 判断依据 |
|----------|-----------|----------|
| 对比类(方案 A vs B，每列是一个方案) | `:::compare` | 标签+说明的二元/多元对比 |
| 流程/步骤(先做 A 再做 B) | `:::flow` 或 `1. [step]` | 有明确顺序的操作或阶段 |
| 时间线/演进历程 | `:::timeline` | 带时间节点的变迁 |
| 核心金句/关键论点 | `> **金句**` | 文章的灵魂观点,值得突出 |
| 补充说明/技术旁注 | `> [!NOTE]` | 不打断正文的附加信息 |
| 注意事项/踩坑提醒 | `> [!WARNING]` | 需要用户警惕的内容 |

### ⛔ 不适合 compare 的场景

- **行×列矩阵对比**（如 SGP 权重表：S/G/P × 普通任务/Deep Research）→ 保留 markdown 表格。compare 的 `tag|title|desc` 三段结构扛不住矩阵，会渲染出错。
- **信息密度高的参数表** → 保留 markdown 表格。

### ⛔ 防重复规则（重要）

组件改写的核心原则：**替换，不是追加**。

金句块、提示块、警告块如果原文已经把同样的意思说清楚了，不要在原文后面再追加一个组件重复一遍。两种正确做法：

1. **原文有概括句 + 金句块重复了**：删掉原文的概括句，只保留金句块（金句块视觉权重更高）
2. **原文已经完整表达**：不要加组件，保持原文

错误做法（会导致内容重复）：
```markdown
架构是现成的，变量是你的领域知识。

> **架构是现成的，变量是你的领域知识。**   ← 重复！
```

正确做法：
```markdown
> **架构是现成的，变量是你的领域知识。**   ← 只保留金句块，删掉重复的原文句
```

**判断原则：**
- 只改写**确实适合**用组件呈现的内容，不强转
- 一篇文章的组件数量控制在 3-6 个，过多反而花哨
- 表格如果信息密度高、不适合卡片化，保留原样
- 不确定是否该用组件的段落，保持原样
- **生成前必须全文扫描：检查每个金句块/提示块/警告块是否与紧邻原文重复**

## 围栏语法硬规则

`:::compare` / `:::flow` / `:::timeline` 围栏必须独占一行，前后都要有空行：

```markdown
正文结束。

:::compare
**A** | 说明
**B** | 说明
:::

下一段正文。
```

错误写法（会导致围栏不被解析，内容以原始 markdown 形式显示）：

```markdown
正文结束。:::compare     ← 没有换行，紧跟正文
**A** | 说明
:::
```

解析正则为 `^:::compare\s*\n([\s\S]*?)\n:::\s*$`，要求 `:::xxx` 在行首。

## 轻量标记使用规则

轻量标记（`==关键词==`）是介于「加粗」和「金句块」之间的中间强调层。

### 三层视觉节奏

| 层级 | 手段 | 频率 | 语法 |
|------|------|------|------|
| 锚点层 | 金句块（居中大字） | 全文 ≤5 处 | `> **核心观点**` |
| 标记层 | 轻量标记（底部细线） | 全文 5-8 处 | `==关键词==` |
| 容器层 | 提示块/警告块 | 按需 | `> [!NOTE]` / `> [!WARNING]` |

### 什么词该标记

- **核心概念首次出现**：`==rule 驱动的主从代理系统==`
- **反直觉判断**：`==不是给你确认的==`
- **关键动作词**：`==先摸底再制定计划==`
- **容易忽略的约束**：`==工具调用预算==`

### 什么词不该标记

- 整句话（那是金句块的工作）
- 高频常见词（「Agent」「任务」「调研」出现太多次不值得标）
- 已经用加粗强调过的词（不要叠加）

### 频率控制

全文标记控制在 **5-8 处**。不是每段都标，只在真正需要读者注意的地方标。标记太多等于没有标记。

## 文章类型→组件配方表

Agent 在分析文章结构时，先判断文章类型，参考以下推荐组件组合作为起点。这不是硬性限制——如果文章实际结构与推荐不符，以文章内容为准。

| 文章类型 | 推荐核心组件 | 推荐点缀组件 | 典型特征 |
|----------|-------------|-------------|----------|
| 技术拆解 | step 步骤 + compare 对比 | WARNING + 轻量标记 | 有明确步骤流程、有对比项、有踩坑点 |
| 案例复盘 | timeline 时间线 | 金句块 + compare | 有时间线演进、有关键判断、有前后对比 |
| 观点判断 | 金句块 | WARNING + compare | 核心是观点输出，少步骤，多判断 |
| 教程操作 | step 步骤 + flow 流程 | NOTE + 轻量标记 | 纯操作流程，有注意事项 |
| 方法论 | 金句块 + compare | step 步骤 | 核心是框架/模型，有对比验证 |

### 使用方式

1. Agent 读完全文后，先判断文章类型
2. 参考配方表选择推荐组件组合
3. 根据文章实际内容微调（增减组件）
4. 控制总组件数在 3-6 个（含轻量标记）
5. 向用户展示改写计划时，说明文章类型判断和组件选择理由

## ⛔ 生成前检查清单（必读）

以下是实战中反复出现的错误。每次生成 HTML 前必须逐条检查，确保不重犯。

### 1. 图片插入位置：必须在完整段落之后，绝不在句子中间

**错误做法**（句号跑到图片后面，或图片打断段落渲染）：
```markdown
注意最后一步——==loop 不是为了让你今天少干活，是为了让系统明天能接着干==

![图片]                          ← 图片占了下一行
。                                ← 句号孤零零留在图片后面
```

```markdown
三个陷阱指向同一件事：==loop 放大的是你，不是 AI==

![图片]。你认真验证，loop 帮你...   ← 图片插在句号前，后续正文被截断
```

**正确做法**（先完成整句话，再独立成段插图）：
```markdown
注意最后一步——==loop 不是为了让你今天少干活，是为了让系统明天能接着干==。

![图片]
```

```markdown
三个陷阱指向同一件事：==loop 放大的是你，不是 AI==。你认真验证，loop 帮你更快地发布可靠代码。

![图片]
```

**规则**：回插图片时，找到目标段落的**最后一个句号**，在句号之后换行插图。绝不能把图片 markdown 插在一个句子中间或句号前面。

### 2. 金句块/提示块/警告块：替换不追加

已在「防重复规则」章节详述。核心：组件是替换原文的概括句，不是在原文后面追加重复内容。

### 3. `:::compare` / `:::flow` / `:::timeline` 围栏：独占一行，前后空行

已在「围栏语法硬规则」章节详述。核心：`:::xxx` 必须行首独立，不能紧跟正文。

### 4. 矩阵对比保留 markdown 表格，不强转 compare

已在「不适合 compare 的场景」章节详述。核心：行×列矩阵（如权重表）用表格，compare 只适合 tag|title|desc 三段结构。

### 5. 横向卡片用 flex:1 等分，不用固定宽度

已在「横向卡片实现方案」章节详述。核心：≤3-4 个卡片用 `flex:1` 等分（内容自动换行），≥4-5 个才加 overflow 滚动。固定宽度 + overflow 会导致长内容把卡片撑太宽。

### 6. `==标记==` 内容可以包含单个 `=` 号

`==六要素 = 心跳==` 这种内容中间有 `=` 号的，正则用 `/(?<!=)==([^\n=][^\n]*?)==/g`，允许中间出现 `=`，只排除 `===` 标题语法。

### 7. 开场动画标题长度检查

标题字符数 × 字号不能超过 viewBox 宽度（640px）。typewriter 模板用 48px 字号时：
- 中文标题 ≤14 字符安全
- 英文标题 ≤22 字符安全
- 超长标题必须缩短，不能硬塞
